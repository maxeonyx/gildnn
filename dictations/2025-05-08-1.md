## Dictation 2 — ML process discipline

"We need some general guidance for machine learning research projects, which includes like a very gradual build-up to any experiment where we have any architecture change needs to be validated on multiple datasets of increasing complexity and difficulty of evaluation. And then every dataset change needs to be manually examined. Every size change needs to go in steps. We don't want to be trying to run a large run without having done all of this rote work, verifying that that large run is actually going to work first."

"Always checkpointing, but not taking too much disk space with checkpoints."

"The very first part of every experiment should simply be to learn the most basic version of the most basic task on the most basic network architecture and prove that that works. So it should be like, you know, five character context next token prediction with a feedforward network or something like that. And then we should iteratively build up the complexity of our tasks and ideally have some kind of regression tests so that we can make changes while maintaining a unified foundation."

"I would like to use UV to manage Python dependencies using a local virtual environment. I'd also like to get all of the system dependencies we need, for example, CUDA versions, these things, installed as soon as possible."

"I would like the code for this repo being kept relatively unified. For example, some kind of abstraction over backbones, over embeddings, over data pipelines, over prediction heads. We should be able to rerun the previous experiments on the unified model architecture. That probably exists already, and I just don't know how to use it. ## Dictation 3 — The vision

"The main idea and motivator is to make a neural network that works like the human brain, as I understand it. What that means is that, first of all, it's actually kind of a recurrent network, but it's still a transformer. Why? Well, in a transformer, the attention is spread over the sequence, and the connections between the input and derived representations of the input along a residual layer, the residual stream, are learned. In this architecture, we have attention, but attention is over many, many unsynchronized recurrent blocks, and there's a graph structure for how they're related to each other. Think of this as the cortex. The neocortex, especially. Each cortical column is a small recurrent block, to start with, just a feedforward block, but in the human brain they are recurrent and they have state. These blocks are tasked with predicting their next time step only of the residual stream at their location. They also receive attention from a central input, a central attention mechanism. The predictive heads are able to draw input from all residual blocks via an attention mechanism, and including raw inputs themselves."

"The inputs that come in every time step are processed by the residual... cortical columns immediately adjacent in the graph, we track the change to the residual stream by those cortical columns. And only when the change is large enough do we propagate that out to additional columns through the graph. And we also propagate it backwards, I believe. There's a bi-directional thing going on here. I'm not sure exactly what that looks like yet."

"While we propagate values across cortical columns, they each learn individually. We unhook the gradients from each piece."

"My goal is then to have all of the different residual columns be unsynchronized, even at the GPU level. Ideally, we would use some kind of volatile memory when communicating between unsynchronized cortical blocks. The point is to speed up the processing, making it embarrassingly parallel."

"We're also effectively going to tune down the rate, although it's going to be ultimately dynamic, of which cortical columns need to be running at a given time step. In fact, the time steps aren't necessarily going to be synchronized. Rather, we're going to have some kind of accumulation and then propagation outwards based on the accumulation, which is based on surprisal. I don't know what that looks like in concrete either."

"The output of a residual block is usable in the attention mechanism or the adjacent blocks in the graph. That node doesn't get recomputed until input from its adjacent nodes or from the global attention triggers it to do so. And that trigger is cheap to recompute. I'm not clear exactly what this looks like either yet."

"Our choice of graph structure should ideally match the GPU architecture rather than the human brain. What's most performant on our GPU as it is? Because I don't think that it's necessarily so important to choose any particular graph structure. The main point is getting unsynchronized blocks so that we improve performance of our prediction part of our network."

## Dictation 4 — Datasets

"A dataset I have great fondness for is a character-by-character English language one. I'm very happy to learn that with our network, then I can talk to it or whatever. See Andre Karpathy's mini-transformer repo."

"Another one I like a lot is what's called arbitrary order sampling image patch dataset. Essentially, we take, we split up an image of any, of various size, we should either be ideally be agnostic over the shape of an image, and we label the image patches as inputs in some way, and we present them to the model in many different orders during training. Randomized orders, raster order, reverse raster, vertical raster, etc. Knight's move, many diverse orderings, including a random ordering as a high proportion of samples. And then a model should get good at predicting image patches, no matter what's next. Then we can query the model for its understanding of images, image patches. No matter what order we want to, you know, if we want to fill in a piece of an image, we can do that. If we want to extend an image outwards, we can."

"It is an RNN, so it's stateful, so we have to think about the inference time usage of our model and how that affects our tasks. For example, we can't just go and relabel all of our inputs and then run the model again without having to reprocess in sequence all of the patches."

"I'm not so interested in arbitrary order of sampling for text. I find that kind of naff, but I am interested in interesting, modern usage of text. For example, in a model like yourself, there is a system prompt, there is a developer prompt, there's a user prompt, and finally, there's actually a conversation. I would be interested in datasets where those are... there are separate spaces of positional encoding, as opposed to being a sequence, they are like, they have some sort of label that says there are different kinds of sequence. And then you could potentially add to, append to the end of the user system prompt during a conversation, rather than appending to the conversation, does that make sense? It's for a self-modifying agent, it could append to its own prompt as opposed to appending to the conversation during the inference, which has a different meaning because the system prompt affects the agent's behavior, whereas the conversation affects it less, it trusts it less. But that's a bit of a digression."

"I'm open to suggestions of other tasks, other more interesting tasks. Probably a image-text unified task would be a great addition."

## Dictation 5 — Mix-Add and noise gate

"I've also got a little idea that you can find on my GitHub somewhere, which is a learned mix-add function for information bottlenecks in models. It's sort of an alignment idea for a measurement system. I'm interested in that, even just the mix function is quite useful for the model in my experience. It helps make training very stable, even if it makes it less performant, I guess. It's simply a way of adding two vectors, but in a principled way that keeps the resulting distribution that if you imagine that the two distributions are normally distributed, then the resulting distribution will have the same norm and still be normally distributed. It's just quite a convenient operation, I found."

[From modularity-loss repo, noise-gating.tex — the actual math:]
mix(a, b, m) = a * sqrt(σ(m)) + b * sqrt(1 - σ(m))
where σ is sigmoid and m is a learnable scalar (or per-channel parameter).
residual(x, f) = mix(x, f(x), m)
Noise gate extension: noise_gate(x, f) = mix(x, mix(noise, f(x), m), m)

"The noise gate is not important. It should be discounted slightly. I'm not, you know, it's just an interesting thing I would like to explore. It's kind of a side quest. It should be integrated into the codebase. I would like a before and after. I would like an ordinary Transformer model to be put into this repo as well."
